🪝 Symlink Attack – Khi đường tắt trở thành cửa hậu

1) Shortcut vs. Symlink

  • Shortcut tạo ra đường tắt tới file thật.
  • Symlink (Linux gọi là symbolic link) cũng vậy, nhưng bản thân nó không chứa nội dung, chỉ trỏ tới một file/thư mục khác.
  • → Có thể coi Symlink là ảnh ảo của file gốc.

Ví dụ Windows gọi là shortcut 🪟, Linux 🐧 gọi là symlink.

# Tạo symlink trong Linux
ln -s /etc/passwd link_to_passwd

2) Các hành vi đặc biệt của Symlink

  • Đọc 👀 / Ghi ✏️ → tác động trực tiếp vào file gốc
  • Xóa ❌ / Di chuyển 🚗 → chỉ ảnh hưởng tới symlink, không ảnh hưởng file gốc
  • Thực thi 🔥 → thật ra là thực thi file gốc
  • Symlink có thể trỏ cả folder, khi ghi file vào folder link → file được ghi trực tiếp vào folder gốc

3) Các viễn cảnh khai thác Symlink

3.1. Giả thuyết 1 – Upload symlink trực tiếp

        Hacker 🕵️
   ┌──────────────────┐
   │  link_to_passwd  │───────┐
   │   (symlink)      │       │ Upload qua Internet 🌐
   └───────┬──────────┘       │
           │                  │
           ▼                  ▼
   ┌──────────────────┐  ┌──────────────────┐
   │   /etc/passwd    │  │ Server 🖥️       |
   │  (file thật)     │  │ link_to_passwd   │
   └──────────────────┘  │(chỉ là “vỏ rỗng”)|  
                         └──────────────────┘

📌 Kết quả: Thất bại 🚫

  • Hacker tưởng sẽ đọc được /etc/passwd trên server.
  • Nhưng thực tế chỉ gửi được file symlink rỗng hoặc server hiểu là file text nhỏ có metadata kiểu vậy, vì symlink không mang nội dung file gốc đi qua Internet.

3.2. Giả thuyết 2 – Đóng gói symlink trong ZIP

Ý tưởng: nén symlink vào .zip rồi upload → khi server giải nén, symlink được tái tạo.

  • zip và unzip của GNU hỗ trợ symlink.
  • Nếu server tự động giải nén → ta có được symlink thật trên server.
  • Thành công đọc file bất kỳ, ví dụ /etc/passwd.

🔎 Ví dụ thực tế: Facebook – link

Hacker: zip -y link_to_passwd step1.zip
Server: unzip step1.zip → tạo lại symlink
Kết quả: đọc được file thật trên server

3.3. Giả thuyết 3 – Từ đọc file → RCE 🚀

Bước nâng cấp:

  1. Tạo symlink trỏ tới thư mục document root (/var/www/html).
  2. Nén lại và upload lên server.
  3. Khi giải nén, symlink “bắc cầu” để ghi file trực tiếp vào webroot.
  4. Upload shell.php → RCE thành công.
[ Hacker ] --upload--> [ step1.zip ]
 → unzip → symlink vào /var/www/html
 → ghi file shell → RCE server

4) Ví dụ thực tế

  • CVE-2018-14364 – Bug trong Ruby RegEx dẫn đến bypass kiểm tra symlink, cho phép RCE.🔗 https://hackerone.com/reports/378148
  • Với Apache: config FollowSymLinks quyết định việc server có đi theo symlink hay không. Nếu bật → nguy cơ đọc/ghi file gốc qua symlink.
<Directory /var/www/>
   Options Indexes FollowSymLinks
   AllowOverride None
   Require all granted
</Directory>

Tóm lại 🎯

Symlink vốn là tính năng hợp pháp, nhưng nếu kết hợp với:

  • Zip/Unzip tự động
  • Sai config (FollowSymLinks)
  • Kiểm tra input lỏng lẻo

→ sẽ biến thành vũ khí nguy hiểm cho attacker để đọc file nhạy cảm hoặc thậm chí RCE.
