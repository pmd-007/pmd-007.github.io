Command Injection

1) Giới thiệu

  • Command Injection là một biến thể của nhóm lỗi Injection: dữ liệu người dùng (input) bị “trộn” vào lệnh hệ điều hành (OS command) và được thực thi ngoài ý muốn.
  • Hậu quả có thể rất nặng: lộ dữ liệu nhạy cảm, tạo backdoor, thậm chí chiếm toàn quyền máy chủ.
  • Tính đến 2025, lỗi này vẫn xuất hiện nhiều ở web UI của thiết bị mạng/IoT (router, NAS, camera), nơi hay có tính năng ping/traceroute/backup chạy lệnh hệ điều hành phía sau.

2) OS command là gì?

Một vài ví dụ lệnh trên Linux:

ls -la
cat /etc/passwd
ping 8.8.8.8

Nếu ứng dụng nối thẳng input của người dùng vào chuỗi lệnh, bạn đã có bệ phóng cho Command Injection.

3) Ví dụ minh họa tối giản

Ứng dụng cung cấp chức năng “ping IP”:

<?php
$ip = $_GET["ip"];
system("ping -c 4 " . $ip);
?>
  • Dev kỳ vọng người dùng nhập 8.8.8.8.
  • Attacker nhập: 8.8.8.8; ls -la

Lệnh thực thi trên server trở thành:

ping -c 4 8.8.8.8; ls -la

👉 Server ping xong sẽ chạy thêm ls -la.
Nếu attacker nhập 8.8.8.8; cat /etc/passwd → rò rỉ thông tin tài khoản hệ thống.

4) Vì sao khai thác được?

Trên shell, có nhiều “toán tử” để nối nhiều lệnh:

  • ; → chạy lệnh 2 bất kể lệnh 1 thành công/thất bại
  • && → chạy lệnh 2 chỉ khi lệnh 1 thành công
  • || → chạy lệnh 2 chỉ khi lệnh 1 thất bại

Ví dụ thử nghiệm:

8.8.8.8; whoami
8.8.8.8 && id
8.8.8.8 || uname -a

5) Pentester workflow (tư duy kiểm thử)

  1. Xác định điểm vào: tham số nào đi tới hàm chạy lệnh (system, exec, popen, shell_exec, ProcessBuilder, v.v.).
  2. Probing: thử payload tối giản ; ls / && ls để xem có bị nối lệnh không.
  3. Xác nhận thực thi: whoami, id, uname -a, đọc file nhỏ.
  4. Mở rộng tác động: đọc config DB, dump source, hoặc leo thang tới RCE toàn hệ thống (tùy phạm vi cho phép trong bài test).

💡 Mẹo: nếu ký tự ; bị chặn, thử đổi sang |, &&, ||, dấu xuống dòng, hoặc cách thoát lồng nhau tùy shell.

6) Thực tế 2025: gặp ở đâu nhiều?

  • Router / NAS / Camera / IoT: giao diện quản trị web có chức năng chẩn đoán mạng (ping, traceroute) hoặc sao lưu—thường gọi command nội bộ.
  • Ứng dụng web nội bộ: trang quản trị tiện ích (convert, backup, scan log, chạy script).
  • CI/CD & tác vụ tự động: hook/build script ghép chuỗi lệnh từ input người dùng.

7) Ví dụ thực tế (tóm tắt ngắn)

  • SaltStack (CVE-2020-11651/11652): chuỗi lỗi dẫn tới thực thi lệnh từ xa trên Salt Master.
  • Cisco RV series / TP-Link / WD MyCloud (nhiều mã lỗi qua các năm): tham số UI không được lọc kỹ → nối vào lệnh hệ điều hành → RCE.
  • Xu hướng đến 2025: các firmware/thiết bị cũ, code tái sử dụng nhiều năm dễ bị Command Injection do thói quen ghép chuỗi lệnh.

(Gợi ý đọc thêm ở phần tham khảo để xem chi tiết từng vụ.)

8) Phòng thủ (nguyên tắc vàng)

Tốt nhất: không dùng shell nếu không thật sự cần.

  • Thay vì system("ping -c 4 " . $ip)
    Hãy dùng API/thư viện native (ví dụ lib ICMP) hoặc thực thi lệnh không qua shell (spawn process với mảng tham số an toàn).
  • Nếu buộc phải gọi lệnh:
    • Whitelisting chặt chẽ đầu vào (ví dụ IP/hostname hợp lệ bằng regex + xác thực thêm).
    • Không cho ký tự đặc biệt của shell (; & | $ ( ) \ ‘ “ < >` …), không cho khoảng trắng thừa.
    • Tách tham số thay vì ghép chuỗi (ngay cả trong các wrapper: execve, ProcessBuilder, subprocess.run([...], shell=False)).
    • Giảm quyền tiến trình (không chạy dưới root; bật AppArmor/SELinux nếu có).
    • Ghi log & monitor lệnh bất thường; bật Alert khi phát hiện chuỗi nghi vấn.

9) Quick checklist cho chính mình

  • Có đang ghép chuỗi lệnh chứa input user?
  • Có check “đúng kiểu dữ liệu” (IP/hostname/port) hay chỉ lọc qua loa?
  • Có cách không dùng shell để đạt cùng mục tiêu?
  • Nếu phải dùng shell: đã chặn ký tự đặc biệt, tách tham số, hạ quyền, bật giám sát?
