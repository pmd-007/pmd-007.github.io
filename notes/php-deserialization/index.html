🐘 PHP Deserialization – Khi dữ liệu hóa trang thành… ác mộng

Mục tiêu: hiểu serialize / unserialize, vì sao PHP Object Injection nguy hiểm, POP chain là gì, và cách phòng thủ.

1) Khởi động nhanh: Object → String → Object

1.1 serialize() là gì?

Chuyển object/array thành chuỗi để lưu trữ hoặc truyền đi (cookie, file, DB…).

<?php
$data = ["name" => "Ash", "pokemon" => "Pikachu"];
$s = serialize($data);
echo $s;
// a:2:{s:4:"name";s:3:"Ash";s:7:"pokemon";s:7:"Pikachu";}

🕹️ Ví dụ đời thường – Save Game: trạng thái nhân vật (HP, tiền, đồ nghề) được “đóng hộp” thành chuỗi rồi ghi vào save.dat.

1.2 unserialize() là gì?

Mở hộp chuỗi → trả lại giá trị/object ban đầu.

<?php
$loaded = unserialize($s);
print_r($loaded);

🕹️ Load Game: đọc save.dat → unserialize() → nhân vật sống lại như cũ.

⚠️ Vấn đề: nếu người dùng điều khiển được chuỗi đầu vào của unserialize(), họ có thể “triệu hồi” object đặc biệt để kích hoạt hành vi ngoài ý muốn.

2) Magic Methods – “cửa bí mật” trong OOP

(OOP = Object Oriented Programming – lập trình hướng đối tượng)

Một số magic methods PHP tự động gọi trong các ngữ cảnh nhất định:

  • __construct() 🏗️: chạy khi khởi tạo object.
  • __destruct() 🧹: chạy khi object bị hủy (cuối request) → “dọn nhà”.
  • __toString() 🗣️: chạy khi object cần in ra chuỗi (echo).
  • __wakeup() ⏰: chạy ngay sau khi unserialize một object.
  • __sleep() 😴: chạy trước khi serialize (chọn thuộc tính để lưu).
  • __invoke() ☎️: chạy khi gọi object như hàm.

Nếu attacker điều khiển thuộc tính của object (property) được tạo ra bởi unserialize(), các magic methods này có thể vô tình gọi code nguy hiểm.

Ví dụ nhỏ – __toString() bất ngờ chạy

class Badge {
  public $level;
  public function __toString() { return "LV-" . $this->level; }
}
function greet($x) { echo "Hello " . $x; } // ép $x thành chuỗi → __toString()
// Nếu $x đến từ unserialize(), attacker có thể set $level = ""...

Ví dụ nhỏ – __wakeup() nguy cơ cao

class Avatar {
  public $url;
  public function __wakeup() {
    // Ý đồ tốt: tải ảnh đại diện khi “hồi sinh”
    file_get_contents($this->url); // Nếu url do user set → SSRF!
  }
}
// unserialize('O:6:"Avatar":1:{s:3:"url";s:18:"http://169.254.169.254/";}');

3) POP – Property Oriented Programming

POP = “lập trình bằng thuộc tính”: attacker set property rồi để magic method tự chạy, tạo thành POP chain (chuỗi gadget).

3.1 Ví dụ code thực tế (tối giản, dễ hiểu)

<?php
class Logger {
  public $filename; // bị attacker kiểm soát
  public function close() {
    // tưởng hiền lành: xóa file log cũ
    @unlink("/var/log/app/" . $this->filename);
  }
}
class Database {
  public $conn;
  public function __destruct() {
    // cleanup cuối request
    if ($this->conn) { $this->conn->close(); }
  }
}

// ❌ Code nguy hiểm
$payload = $_POST['p'] ?? '';
$obj = unserialize($payload); // user-controlled!

Kịch bản tấn công:

  1. Attacker tạo object Logger với $filename = "../../www/index.php" (Path Traversal).
  2. Nhét Logger vào property $conn của Database.
  3. Khi request kết thúc → Database::__destruct() gọi $conn->close() → Logger::close() chạy → unlink file nhạy cảm.

Payload minh họa (mô phỏng, tùy phiên bản PHP/class):

O:8:"Database":1:{s:4:"conn";O:6:"Logger":1:{s:8:"filename";s:24:"../../www/index.php";}}

3.2 Flow minh họa (domino đổ)

User input (serialized)
        ↓
unserialize()  →  tạo object + property theo ý attacker
        ↓
magic method kích hoạt (wakeup / destruct / toString)
        ↓
gadget 1  →  gadget 2  →  gadget 3 …
        ↓
File delete / SSRF / LFI / RCE (Remote Code Execution)

Liên hệ vui: ghép lego từ các mảnh class sẵn có → tạo robot phá cửa. Dev không đưa robot, nhưng vô tình đưa đủ mảnh lego.

4) Thêm vài “món” dễ dính lỗi

4.1 Cache/Session tự chế bằng serialize

// Lưu session vào DB
$db->save("sess_{$uid}", serialize($cart));
// Khôi phục
$cart = unserialize($_POST['cart'] ?? ""); // ❌ user-controlled → toang

👉 Dùng json_encode()/json_decode(true) an toàn hơn (chỉ mảng/chuỗi/số).

4.2 Log/Queue/Job có hành vi file/exec

class Job {
  public $cmd;
  function __destruct() {
    if ($this->cmd) { system($this->cmd); } // ❌
  }
}
// Payload set $cmd="id" → RCE

4.3 Tự động tải tài nguyên trong __wakeup()

Đừng gọi HTTP/FS trong __wakeup() với dữ liệu không kiểm soát → dễ SSRF/LFI.

5) Phòng thủ – Làm sao để không “toang”

5.1 Tránh unserialize() với dữ liệu không tin cậy

  • Thay bằng JSON: 
    $safe = json_decode($_POST['data'] ?? "{}", true, 512, JSON_THROW_ON_ERROR);
  • Ưu điểm: chỉ tạo array/scalar, không sinh object.

5.2 Nếu bắt buộc phải unserialize

  • PHP 7+: dùng allowed classes để chặn tạo object lạ: 
    $obj = unserialize($payload, ['allowed_classes' => ['MyDto','SafeClass']]);
  • Tắt hoặc tránh logic nguy hiểm trong __wakeup() / __destruct().
  • Với file path: dùng allowlist, chặn ../, chuẩn hóa đường dẫn thật (realpath).

5.3 Code hygiene

  • Không thực thi shell (system/exec/shell_exec/passthru) với dữ liệu chưa kiểm soát.
  • Không thao tác file nhạy cảm trong cleanup (destructor).
  • Review class nào có side-effect khi khởi tạo/hủy → cẩn thận khi nó có thể được tạo từ unserialize.

5.4 Phát hiện trong thực tế (từ góc pentest)

  • Tìm chỗ app xử lý cookie, form, hidden field, header có chuỗi kiểu a:1:{...}.
  • Thử payload vô hại trước (để dò __wakeup()):O:4:"Test":0:{} (class tồn tại → báo lỗi/trace khác thường?).
  • Tận dụng lỗi/exception để hiểu chain; xây PoC nhỏ → lớn (domino).

6) Checklist nhanh

  • Không dùng unserialize() với input từ user.
  • Nếu dùng → allowed_classes + validate kỹ.
  • Tránh side-effect nguy hiểm trong __wakeup() / __destruct() / __toString().
  • Dùng JSON cho dữ liệu cấu trúc trao đổi giữa client–server.
  • Kiểm soát đường dẫn file (allowlist, realpath, chặn ..).
  • Hạn chế quyền process (không cần root), tách quyền đọc/ghi.

7) Từ điển mini (viết tắt gặp nhiều)

  • OOP: Object Oriented Programming – lập trình hướng đối tượng.
  • POP: Property Oriented Programming – khai thác dựa trên thuộc tính.
  • RCE: Remote Code Execution – thực thi mã từ xa.
  • LFI/RFI: Local/Remote File Inclusion – đọc/nhúng file cục bộ/từ xa.
  • SQLi: SQL Injection – chèn câu lệnh SQL.
  • PoC: Proof of Concept – mẫu chứng minh khai thác.

8) Lời kết (hơi hài nhưng thật)

“Đừng để user viết nhật ký hộ bạn. Nó sẽ không ghi hôm nay trời đẹp, mà sẽ ghi: rm -rf /.” 😅 Giữ object của bạn hiền hòa bằng cách không unserialize dữ liệu lạ. Khi buộc phải dùng – hãy xích nó bằng allowed_classes, rào bằng JSON, và dọn nhà (destructor) một cách an toàn.
