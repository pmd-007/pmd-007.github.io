🕵️ Black-box Testing – Pentest khi không có đèn pin

1) Tổng quan về các phương thức pentest

Pentest có nhiều “trường phái” tùy vào lượng thông tin pentester có trong tay:

White-box  [ ██████ ]   → Có nhiều info (source code, config, account test…)
Gray-box   [ ███░░░ ]   → Có một phần info (VD: chỉ có user test)
Black-box  [ ░░░░░░ ]   → Không có info, chỉ có URL → đúng chất hacker ngoài đời

Thông tin có thể là:

  1. Source code
  2. Kiến trúc hệ thống / network topology
  3. Cấu hình (config)
  4. Testing account (user, admin)
  5. Môi trường (docker, debug setup…)

2) Ưu & nhược của White-box vs Black-box

White-box (thấy rõ mê cung bên trong)

Input →
┌───────────────────────┐
│ ┌───┐ ┌─────┐ ┌─────┐ │
│ │   │ │ ┌─┐ │ │     │ │
│ │ ┌─┘ │ │ │ │ └─┐ ┌─┘ │
│ │ │   │ │ │ │   │ │   │
│ │ │ ┌─┘ │ └─┘ ┌─┘ └─┐ │
│ │ │ │   │     │     │ │
│ │ └─┘ ┌─┘ ┌───┘ ┌───┘ │
│ │     │   │     │     │
│ └─────┘ ┌─┘ ┌───┘ ┌───┤
│         │   │     │ O │ → Output
└─────────┴───┴─────┴───┘
  • Có nhiều (hoặc toàn bộ) thông tin
  • Quyền truy cập lớn → debug, đọc log
  • Dò lỗi nhanh, quét kỹ được logic bên trong

Black-box (hộp đen kín mít)

Input →
░░░░░░░░░░░░░░░░░░░░░
░░░░░░░░░░░░░░░░░░░░░
░░░░░░░░░░░░░░░░░░░░░
░░░░░░░░░░░░░░░░░░░░░
░░░░░░░░░░░░░░░░░░░░░
→ Output
  • Không có info
  • Bị hạn chế quyền, phải “thử & sai”
  • Dựa vào quan sát kết quả → đoán bug

👉 Tóm gọn: Black-box giống như đưa input bất kỳ vào ứng dụng rồi quan sát output để đưa ra “dự đoán có học” (educated guess) xem có bug hay không.

Ví dụ:

?id=1337'
→ ERROR: SQL syntax

→ Có mùi SQLi ngay 👀

3) Black-box Testing giống gì với khám bệnh 🩺

Black-box testing cũng giống bác sĩ chẩn đoán bệnh:

  • Không thấy bên trong cơ thể → chỉ dựa vào triệu chứng & dấu hiệu
  • Thiếu thông tin → phải dựa vào kinh nghiệm + xét nghiệm để kết luận
Triệu chứng → [ 👨‍⚕️ Bác sĩ ] → Kết luận
   Input     → [ Pentester  ] → Bug ?

4) Quy trình 4 bước Black-box Testing

1) Đánh giá sơ bộ
   - Liệt kê chức năng: login, upload, logging...
   - Đoán bug có thể: login → SQLi, upload → file upload, logging → command injection
       ↓
2) Giả thuyết & quan sát dấu hiệu
   - Input → quan sát lỗi, data lạ, time delay...
       ↓
3) Kiểm tra chi tiết (xét nghiệm)
   - Payload cụ thể → kiểm chứng giả thuyết
       ↓
4) Kết luận sơ bộ
   - Có bug → note lại
   - Không → quay lại bước 2

5) Dấu hiệu cần chú ý 👀

Khi test Black-box, hãy soi:

  • Error messages (lỗi SQL, stack trace…)
  • Unexpected data (trả về thêm thông tin lạ)
  • Time response bất thường (delay → có thể time-based injection)
  • Unexpected behavior (app hoạt động khác so với bình thường)

6) Một số Tip & Trick

  • Luôn để ý untrusted data: GET/POST params, cookie, filename…
  • Cẩn thận với dấu hiệu mờ nhạt, dễ nhầm với bug khác
  • Đừng chỉ nhìn bug → hãy nhìn bức tranh tổng thể input/output
  • Hack là khai thác hành vi → hiểu hành vi trước thì mới tìm bug chuẩn
  • Chức năng phức tạp thì take note, có thể quay lại sau
  • Yếu tố may mắn cũng đóng vai trò khi làm black-box 😅

7) Đúc kết 📝

Black-box Testing là khi:

  1. Không có thông tin, chỉ có input/output
  2. Quan sát kỹ để tìm ra dấu hiệu bất thường
  3. Đặt giả thuyết bug → kiểm chứng bằng payload
  4. Ghi chú & loại trừ trường hợp sai
  5. Lặp lại cho đến khi chắc chắn
Input → [ Hộp đen ??? ] → Output
         ↑    quan sát   ↓
        giả thuyết ← test
